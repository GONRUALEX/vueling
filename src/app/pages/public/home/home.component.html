Bienvenid@ a la prueba técnica de Vueling

A continuación expongo como está hecho el código y las razones escogidas.

HTML

Todo el html está hecho semánticamente ayudando a la mejora de la accesibilidad, SEO, mantenimiento del código, etc.

CSS

Para el css se ha seguido una metodología BEM, aprovechando las características de scss. Se han creado en la raíz archivos para las fuentes, colores de la aplicación y se han definido como primary y accent para angular material los colores proporcionados por la primera versión del código para estandarizar la parte visual.

Errores

Los errores se han centralizado, se usa un interceptor para su captura y su estado se maneja en un servicio.

Initzializer

Para el manejo y persistencia de la sesión del usuario una vez iniciada la sesión al introducir usuario y contraseña se ha utilizado el localStorage manejado por un servicio, el inizializer es el encargado de comprobar si existe la key cuando se refresca la aplicación o se entra por primera vez.

Servicios

Para el manejo del estado de la aplicación se han utilizado varios servicios.

Estructura

La estructura se compone de 4 carpetas principales, expongo una breve descripción:

  • Core: contiene servicios, interceptores, guards, initzializer... Es importante que se manejen los determinados elementos usando el patrón singleton para que exista una única instancia de estos.
  • Layout: contiene los componentes estáticos de la aplicación, entre ellos header y footer.
  • Shared: contiene todos los elementos compartidos por toda la aplicación.
  • Pages: contiene las páginas de la aplicación.

Modulos

La aplicación está modularizada, cada módulo se encarga de cargar los elementos de cada parte de la aplicación. Para las páginas se hace una carga perezosa de los módulos.

Alias

Se han añadido Alias para cada una de las carpetas.

Archivo config

Existe un archivo config encargado de tener todas las llamadas a la api, de esta manera se centraliza y controla todas las llamadas http de la aplicación.

Tipado

Todos los datos se han tipado convenientemente.

Constantes

Todas las constantes de la aplicación se han declarado en un archivo constants en la carpeta utils, de esta manera se evita hacer match a cadenas de texto sueltas, en otros casos se han utilizado tipos enumerados dependiendo de la casuistica.

Environments

Existe un archivo general para los environments, de esta manera para las propiedades compartidas para los entornos no repetiremos tantas claves valor.

Llamadas Http

Se ha creado una clase generica para poder reaprovecharla en la aplicación en el caso que existieran otras llamadas con otros tipos de datos.

Notificaciones

Hay un sistema de notificaciones que utiliza angular material para sacar los mensajes tanto en caso de error como a la hora de saludar y despedirse en el momento de logearse o cerrar la sesión.

Ejecutar en local aplicación

Para ejecutar en local la aplicación existe un script como se comentaba en la documentación, para ejecutar npm run start:all:windows en el caso de windows y npm run start:all:linux en el caso de linux, se abrirán dos terminales, una con la ejecución del servidor angular y otra con el servidor stubs.

Detalle aeropuerto

Para ver el detalle del aeropuerto solo hay que pulsar en cualquiera de ellos, esto abre una modal con la información.

Test

Se han completado los test de varios servicios y componentes.

Accesibilidad

Además de html semántico se han añadido atributos ARIA para mejorar la accesibilidad.

Performance

La performance de la página a tratar tienes estos valores:

  • First Contentful Paint : 1.1 s
  • Largest Contentful Paint : 2.0 s
  • Total Blocking Time : no hay javascript bloqueante
  • Cumulative Layout Shift : 0.004, hay algún layout shift se debería mirar con más calma con una conexión 3g para corregirlo
